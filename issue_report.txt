----------------------------

TEST PROCESS:

2. Delete the user from the 'Users' management page.
3. Check the 'servers' table in the database.
4. EXPECTED RESULT: Both 'users.deleted_at' and 'servers.deleted_at' should be set, but verify if the logic in 'UserRepository::delete' and 'ServerRepository::delete' causes issues.

8. FIXED: INSECURE GET DELETIONS & CSRF
-------------------------------------
Status: RESOLVED

Many delete buttons were using 'GET' requests (window.location.href), which is insecure and bypassed CSRF protection. This often caused "broken" behavior where actions wouldn't process correctly.

FIXED IN: 
- 'ScheduleController.php', 'ServerController.php', 'AnnouncementController.php'
- Updated views to use hidden 'POST' forms with 'csrf_field()' for all delete actions.
- Added 'verify_csrf()' to the corresponding controller methods.

-----------------------------------
Status: RESOLVED
Fixed a bug in 'Schedules' view where bulk delete was pointing to a non-existent '/schedules/bulk-action' route. It now correctly points to '/schedules/bulk-delete'.
